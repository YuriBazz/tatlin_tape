# Tatlin Test (Tape sorting)
Тестовое задание для yadro impulse в команду tatlin.datapath

# Stack
``C++ 23`` ``g++ 15`` ``cmake >= 4.1`` ``make 4.4.1``

# Build
```bash
mkdir build
cmake -B build -DCMAKE_CXX_FLAGS=""
cd build && make
```
Флаги, которые могут пригодиться (на самом деле только дефайны):

``-DRANDOM_TEST`` — добавляет тест со случайной генерацией наполнения файлов; делается 100 запусков;
каждый файл содержит 1e5 записей; размер ram генерируется случайно; тест содержит примитивный бенчмарк.

``-DSLEEPS`` — включает имитацию задержки при движении указателя считывателя.

# Run

Проект состоит из:

``tatlin_tape_tests`` — тесты.

``tatlin_tape_core`` — утилита для tape sorting.

``generator`` — генератор бинарного файла, заполненного int. Генератор может создавать файлы, если их нет по пути назначения, 
но не производит проверку на существование пути. Подробнее в ``Принятых решениях``.

``checker`` — проверяет, что файл был правильно отсортирован. Подробнее в ``Принятых решениях``.

Для запуска необходимо, находясь в директории ``build``, запустить соответственно

```
./test/tatlin_tape_tests
```

Необходимыми являются только пути до источника и приемника.
```
./core/tatlin_tape_core -s <path_to_src> -d <path_to_dst> -r <size_of_ram_in_bytes> -c <path_to_config.json>
```

```
./generator/generator -p <path_for_generated_file> -c <count_of_generated_ints>
```

```
./checker/checker -o <path_to_original_file> -s <path_to_sorted_file>
```

# Принятые решения

### Структура 

Т.к. в задании не был указан способ представления ленты в памяти, было принято решение считать, что лента — бинарный файл, заполненный int.

Отсюда, у меня возникла необходимость быстро эти файлы получать. Поэтому в проекте есть генератор и проверяльщик.
Если будете пользоваться генератором, обратите внимание, что он стирает файл.

Далее ничего не было сказано про файл конфигурации. Было принято решение считать, что это будет ``*.json`` файл, 
для чего была подключена header-only библиотека для парсинга.

Пример файла конфигурации:
```json
{
  "reading_delay" : 11124812421,
  "writing_delay" : 12841241,
  "shift_delay" : 2032590125123,
  "rewind_delay" : 48213491239
}
```
Было принято решение считать, что задержки тут — число микросекунд.

Далее ничего не было сказано про сами задержки. Было принято решение просто использовать ``this_thread::sleep_for()`` для их имитации.

Было принято решение считать, что считыватель — неотъемлемая часть ленты, а будущее ограничение на кол-во лент 
было воспринято, как ограничение на число используемых при сортировке файлов, в ограничениях, что написанное на ленте нельзя стереть, а только затереть.

### Алгоритм

Был реализован tape sorting на ленте-приемнике и двух дополнительных временных лентах.
Утверждение, что было использовано ровно две дополнительные ленты, считаю справедливым, т.к. ограничение возможности отчистки ленты весьма существенно.

Было принято решение оставить ram_size отвечать только за кол-во считываемых с ленты значений, а сортировщику позволить использовать несколько больше памяти.
Такое решение кажется несущественным, т.к. сам ram_size — не константная величина.
